Create `firestore.rules` in the project root. CRITICAL: subcollection rules do NOT inherit from parent rules — each subcollection needs its own explicit match block. The purchaseStatus privacy boundary (D-03) is the single most important correctness constraint.

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Helper functions
    function isAuthenticated() {
      return request.auth != null;
    }

    function isOwner(wishlistId) {
      return isAuthenticated() &&
             get(/databases/$(database)/documents/wishlists/$(wishlistId)).data.childUid == request.auth.uid;
    }

    function isViewer(wishlistId) {
      return isAuthenticated() &&
             request.auth.uid in get(/databases/$(database)/documents/wishlists/$(wishlistId)).data.viewerUids;
    }

    // Root wishlist document — child (owner) can read/write their own
    match /wishlists/{wishlistId} {
      allow read: if isAuthenticated() && (isOwner(wishlistId) || isViewer(wishlistId));
      allow create: if isAuthenticated();
      allow update, delete: if isOwner(wishlistId);

      // Items subcollection — child-readable AND child-writable (D-02)
      // Viewers can also read items (to show the wishlist to them)
      match /items/{itemId} {
        allow read: if isAuthenticated() && (isOwner(wishlistId) || isViewer(wishlistId));
        allow write: if isOwner(wishlistId);
      }

      // Purchase status subcollection — VIEWER-ONLY, child CANNOT read (D-03)
      // This is the irreversible privacy boundary. Child UID is NOT in viewerUids.
      // Do NOT add any rule that allows childUid to read this subcollection.
      match /purchaseStatus/{itemId} {
        allow read, write: if isViewer(wishlistId);
      }
    }

    // Username → UID mapping (D-04)
    // Anyone authenticated can read (needed for login lookup)
    // Only the Admin SDK (server-only) writes these — no client write rule needed
    match /usernames/{username} {
      allow read: if isAuthenticated();
      allow write: if false; // Admin SDK only
    }

    // User profiles (D-04)
    // Users can read/write their own profile; others cannot
    match /users/{uid} {
      allow read, write: if isAuthenticated() && request.auth.uid == uid;
    }

    // Invite tokens (D-04)
    // Share link redemption via API Route + Admin SDK only — client SDK never reads invites
    match /invites/{token} {
      allow read, write: if false; // Admin SDK only, client never touches this
    }
  }
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Firestore | Client SDK makes direct calls to Firestore; security rules are the only enforcement layer |
| server → Firestore | Admin SDK bypasses all security rules; only used in API routes guarded by server-only |
| env file → git | .env.local contains Firebase client config; .env files must not be committed |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Elevation of Privilege | firestore.rules / purchaseStatus subcollection | Critical | mitigate | Explicit `match /purchaseStatus/{itemId}` block with `allow read, write: if isViewer(wishlistId)` — child UID (not in viewerUids) is denied by default. Verified by emulator test in Plan 02. |
| T-01-02 | Information Disclosure | firestore.rules / purchaseStatus subcollection | Critical | mitigate | No rule grants childUid access to purchaseStatus. Subcollection rules are explicit (not inherited from parent). Emulator test in Plan 02 assertFails for child UID. |
| T-01-03 | Information Disclosure | .env.local / admin credentials | High | mitigate | .env.local added to .gitignore. FIREBASE_PRIVATE_KEY uses plain env var (not NEXT_PUBLIC_ prefix). .env.example committed instead with placeholder values. |
| T-01-04 | Elevation of Privilege | lib/firebase/admin.ts (created in Plan 02) | High | mitigate | import 'server-only' at top of admin.ts — build fails if accidentally imported in client code. Admin SDK never exposed to browser bundle. |
| T-01-05 | Spoofing | usernames/{username} collection | Medium | mitigate | Rule: `allow write: if false` — only Admin SDK can create username→uid mappings, preventing username squatting via client SDK. |
| T-01-06 | Elevation of Privilege | invites/{token} collection | Medium | mitigate | Rule: `allow read, write: if false` — share link redemption only via API Route + Admin SDK (enforced in Phase 4). Client SDK cannot read or enumerate invite tokens. |
| T-01-07 | Tampering | firestore.rules / wishlists document | Medium | mitigate | `allow update, delete: if isOwner(wishlistId)` — only child who owns the wishlist can modify the wishlist document. Viewer cannot tamper with childUid or viewerUids fields. |
| T-01-08 | Denial of Service | Firestore / open write rules | Low | accept | No rate-limiting in Firestore rules for Phase 1. Firestore free tier has built-in quotas. Rate-limiting can be added in Phase 5 hardening if needed. |
</threat_model>
