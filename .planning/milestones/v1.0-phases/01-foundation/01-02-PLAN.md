Create `src/lib/firebase/client.ts` with the following exact content. This is the client-side Firebase singleton used in all browser components.

```typescript
import { initializeApp, getApps, getApp } from 'firebase/app';
import { getFirestore } from 'firebase/firestore';
import { getAuth } from 'firebase/auth';

const firebaseConfig = {
  apiKey: process.env.NEXT_PUBLIC_FIREBASE_API_KEY,
  authDomain: process.env.NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN,
  projectId: process.env.NEXT_PUBLIC_FIREBASE_PROJECT_ID,
  storageBucket: process.env.NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET,
  messagingSenderId: process.env.NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID,
  appId: process.env.NEXT_PUBLIC_FIREBASE_APP_ID,
};

// Guard: only initialize if not already done.
// Required for Next.js HMR — without this guard, every hot-reload calls initializeApp()
// again and throws "Firebase app already exists [DEFAULT]".
const app = getApps().length === 0 ? initializeApp(firebaseConfig) : getApp();

export const db = getFirestore(app);
export const auth = getAuth(app);
```

Notes:
- `auth` is exported alongside `db` — Phase 2 needs it for login/logout flows. Adding it now costs nothing.
- Do NOT add `'use client'` at the top — this is a utility module, not a React component. Client components that import it will be marked `'use client'` themselves.
- Do NOT import `server-only` here — this file IS the client module.

After creating the file, run `npm run build` to confirm no TypeScript errors.

Create `src/lib/firebase/admin.ts` with the following exact content. `import 'server-only'` MUST be the very first line — it is a build-time guard that causes Next.js to throw a build error if this file is ever imported in a client component.

```typescript
import 'server-only';

import { initializeApp, getApps, getApp, cert } from 'firebase-admin/app';
import { getFirestore } from 'firebase-admin/firestore';
import { getAuth } from 'firebase-admin/auth';

const adminApp =
  getApps().length === 0
    ? initializeApp({
        credential: cert({
          projectId: process.env.FIREBASE_PROJECT_ID,
          clientEmail: process.env.FIREBASE_CLIENT_EMAIL,
          // env vars store the private key with literal \n characters — replace them with real newlines
          privateKey: process.env.FIREBASE_PRIVATE_KEY?.replace(/\\n/g, '\n'),
        }),
      })
    : getApp();

export const adminDb = getFirestore(adminApp);
export const adminAuth = getAuth(adminApp);
```

Notes:
- `adminAuth` is exported alongside `adminDb` — Phase 2 uses `adminAuth.setCustomUserClaims()` to assign child/viewer roles.
- The `privateKey` `replace(/\\n/g, '\n')` call converts the literal `\n` escape sequences that appear when storing a PEM key in an environment variable back into real newlines. Without this, Firebase Admin will reject the credential with an "invalid PEM" error.
- Imports come from `firebase-admin/app`, `firebase-admin/firestore`, and `firebase-admin/auth` (sub-package imports), NOT from `firebase-admin` (the legacy compat entry). This is the current modular approach.

After creating the file, run `npm run build` to confirm no TypeScript errors. The `server-only` import will cause an intentional build error ONLY if the file is imported from a client component — a plain build with no imports of this file will succeed.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client component → admin.ts | If admin.ts is imported from a client component, admin credentials ship to the browser |
| env var → admin credential | FIREBASE_PRIVATE_KEY must never reach the browser bundle |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Information Disclosure | src/lib/firebase/admin.ts in browser bundle | Critical | Mitigate | `import 'server-only'` on line 1 of admin.ts causes a Next.js build error if the file is imported in any client component. Prevents Admin SDK (with full database access) from ever reaching the browser. |
| T-02-02 | Elevation of Privilege | Admin SDK bypasses Firestore security rules | High | Accept | Admin SDK is intentionally privileged. Mitigation is scoping: only API routes and Server Components may import admin.ts. The `server-only` guard enforces this at build time. |
| T-02-03 | Information Disclosure | FIREBASE_PRIVATE_KEY in env | Critical | Mitigate | Admin env vars use NO `NEXT_PUBLIC_` prefix — Next.js strips these from the client bundle entirely. They are only available server-side. |
| T-02-04 | Tampering | getApps() guard allows multiple apps | Low | Accept | The guard `getApps().length === 0 ? initializeApp(...) : getApp()` prevents duplicate initialization. A race condition where two simultaneous requests both see 0 apps is handled by Firebase's internal initialization mutex. |
</threat_model>
