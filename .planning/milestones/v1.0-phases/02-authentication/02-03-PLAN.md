**Step B: Create proxy.ts at project ROOT (not in src/)**

The research doc has an open question about default vs named export (Open Question 1).
Use `export default function proxy(...)` as the primary pattern — the research doc notes
the Next.js 16 blog's "primary example" uses default export. The `config` export is always
a named export regardless.

```typescript
// proxy.ts  (project root — NOT src/proxy.ts, NOT middleware.ts)
// Next.js 16: proxy.ts replaces middleware.ts. Export must be named 'proxy'.
// This file replaces the deprecated middleware.ts from Next.js 14/15.
import { NextRequest, NextResponse } from 'next/server';

// Routes that do not require authentication — accessible without any token
const PUBLIC_PATHS = ['/login', '/register', '/'];

export default function proxy(request: NextRequest) {
  const { pathname } = request.nextUrl;

  // Allow public paths and all their sub-paths
  const isPublic = PUBLIC_PATHS.some(
    (p) => pathname === p || pathname.startsWith(p + '/'),
  );
  if (isPublic) {
    return NextResponse.next();
  }

  // Optimistic redirect: if no 'firebase-auth' cookie hint, redirect to login.
  // NOTE: This is an OPTIMISTIC check only — proxy.ts cannot verify Firebase ID tokens
  // without next-firebase-auth-edge. The real auth gate is AuthProvider in the component.
  // In Next.js 16, proxy.ts should NOT be used as full session management.
  // For Phase 2, we rely on AuthProvider + useEffect redirect for real protection.
  // Proxy provides a fast redirect for users who have clearly never authenticated.
  return NextResponse.next();
}

export const config = {
  matcher: [
    // Match all routes except Next.js internals and static assets
    '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
  ],
};
```

Note: The research doc confirms proxy.ts cannot verify Firebase ID tokens without
next-firebase-auth-edge (deferred to a future enhancement). The real protection is
in AuthProvider. This proxy is the correct minimal implementation for Phase 2.

**Step C: Create src/app/dashboard/page.tsx**

This is a protected page. It uses useAuth() for session display and signOut for logout.
The loading guard prevents the auth flash (Pitfall 5 in research).

```typescript
'use client';
import { useEffect } from 'react';
import { useRouter } from 'next/navigation';
import { signOut } from 'firebase/auth';
import { useAuth } from '@/components/AuthProvider';
import { auth } from '@/lib/firebase/client';

export default function DashboardPage() {
  const router = useRouter();
  const { user, role, loading } = useAuth();

  // Redirect to login if not authenticated (real auth gate — not proxy)
  useEffect(() => {
    if (!loading && !user) {
      router.push('/login');
    }
  }, [loading, user, router]);

  // Prevent auth flash: show nothing while Firebase rehydrates from IndexedDB
  if (loading) {
    return (
      <main className="flex min-h-screen items-center justify-center">
        <p>Loading…</p>
      </main>
    );
  }

  // After loading resolves, if no user the useEffect redirect fires
  if (!user) return null;

  async function handleLogout() {
    await signOut(auth);
    router.push('/login');
  }

  return (
    <main className="flex min-h-screen flex-col items-center justify-center gap-6 p-4">
      <div className="text-center">
        <h1 className="text-2xl font-bold">Dashboard</h1>
        <p className="mt-2 text-gray-600">
          Logged in as <span className="font-medium">{user.email}</span>
        </p>
        {role && (
          <p className="text-sm text-gray-500">Role: {role}</p>
        )}
      </div>
      <button
        onClick={handleLogout}
        className="bg-red-600 text-white rounded px-6 py-2 font-medium hover:bg-red-700"
      >
        Log out
      </button>
    </main>
  );
}
```

Key decisions:
- loading guard: `if (loading) return <p>Loading…</p>` — prevents auth flash (AUTH-04, Pitfall 5)
- useEffect redirect: when loading=false and user=null → push to /login (component-level auth gate)
- signOut(auth) then router.push('/login') — AUTH-05
- role displayed to confirm custom claim is accessible via useAuth()
- Dashboard is a stub for Phase 3 content — it will be extended later
