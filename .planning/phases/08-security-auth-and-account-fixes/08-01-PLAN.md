<objective>
Tighten Firestore security rules and add HTTP security headers. This plan closes the two highest-severity live exploit vectors: an overly permissive wishlist create rule that lets any authenticated user forge wishlist docs, and the complete absence of security headers that amplifies XSS risk.

Purpose: Eliminate privilege escalation via forged Firestore documents (SEC-01) and add defence-in-depth against clickjacking and MIME confusion (SEC-03). The getOrCreateWishlist companion fix (DEBT-07) is a required companion to SEC-01 — without it the tightened rule breaks child first-login.
Output: Updated firestore.rules, updated next.config.ts with headers(), updated getOrCreateWishlist in wishlist.ts.
</objective>
