## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, drag handle inner padding, price/label inline spacing |
| sm | 8px | Field label-to-input gap, card internal row gaps, thumbnail margin |
| md | 16px | Card padding (all sides), form field vertical spacing, page horizontal padding |
| lg | 24px | Gap between cards in the list, section padding above/below list |
| xl | 32px | Page top padding, gap between page header and list |
| 2xl | 48px | Empty state vertical centering offset |
| 3xl | 64px | Not used in this phase |

Exceptions:
- Touch targets for drag handle and action buttons: minimum 44px tall (iOS HIG compliance) — achieved via `min-h-[44px]` on interactive elements
- Inline-edit form fields: `py-2` (8px vertical) to match existing login/register form pattern

**Source:** 8-point grid default; touch-target exception per mobile usability requirement (UI-02, D-15).

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px (`text-base`) | 400 (regular) | 1.5 | Item note text, form field values, descriptive copy |
| Label | 14px (`text-sm`) | 500 (medium) | 1.4 | Field labels, price display, URL display, card metadata |
| Heading | 20px (`text-xl`) | 600 (semibold) | 1.2 | Item title on card, page heading ("Din önskelista") |
| Display | 28px (`text-3xl`) | 700 (bold) | 1.2 | Empty state heading only |

**Weights used:** regular (400) + semibold (600). Medium (500) used only for label role — this is a strict two-weight system in practice; medium is a label-only exception to avoid visual noise.

**Source:** Existing pages use `text-2xl font-bold` and `text-sm font-medium` — Phase 3 formalizes and extends this scale with a 4-step system. Display size added for empty state emotional impact (D-14).

## Color

This phase establishes the pastel foundation. Phase 5 will refine exact hue values — the tokens below must be defined as CSS custom properties so Phase 5 can update them in one place.

| Role | Value | Tailwind Class | Usage |
|------|-------|----------------|-------|
| Dominant (60%) | `#FFF9F5` | `bg-[#FFF9F5]` (override background CSS var) | Page background, wishlist surface |
| Secondary (30%) | `#FFF0E8` | `bg-[#FFF0E8]` | Item cards, inline-edit form background, skeleton loaders |
| Accent (10%) | `#F97316` (warm orange-rose) | `bg-[#F97316]` / `text-[#F97316]` | Reserved for: primary Add CTA button, Save button in edit mode, drag handle on active drag state |
| Destructive | `#DC2626` | `text-red-600` / `bg-red-600` | Delete button label and inline confirmation text only |
| Text primary | `#171717` | `text-[#171717]` (existing foreground var) | All body and heading text |
| Text muted | `#6B7280` | `text-gray-500` | Price, URL, metadata labels, placeholder text |
| Border | `#E5D5CC` | `border-[#E5D5CC]` | Card border, form field border, dividers |

**Accent reserved for:** primary CTA "Lägg till önskemål" button, Save button during inline edit, drag handle highlight during active drag. No other elements use the accent color.

**Color rationale:** Warm off-white dominant (#FFF9F5) with a slightly deeper warm card surface (#FFF0E8) delivers the pastel, child-friendly feel required by UI-01 without being gendered. Orange accent is warm and playful — gender-neutral and compatible with a child audience. Phase 5 will audit and may adjust hue ±10° based on full palette review.

**Source:** Claude's discretion (CONTEXT.md), constrained by UI-01 (pastel, gender-neutral, child-friendly). Phase 5 will polish.

## Component Inventory

These are the UI primitives Phase 3 must implement. No library — built from Tailwind.

### WishlistPage (`/wishlist`)
- Auth guard (same pattern as dashboard: `useEffect` + `router.push('/login')`)
- Loading skeleton while Firestore query resolves
- Page heading: "Din önskelista" at Heading scale
- Add-item trigger button (always visible, bottom of list or top when list is empty)
- Real-time Firestore listener on `wishlists/{id}/items` ordered by `position`

### WishItemCard
- Wide card layout — all fields visible (D-05)
- Thumbnail image (64×64px, `object-cover`, `rounded-md`) when `imageUrl` set; placeholder rectangle (same size, `bg-[#E5D5CC]`) when absent (D-06)
- Title at Heading scale (20px semibold)
- Price at Label scale (14px medium) prefixed with "~" when set
- Product URL at Label scale, truncated to one line, rendered as `<a>` with external link behavior
- Note at Body scale (16px regular), max 3 lines with line clamp
- Drag handle: always-visible 24×44px grip area on left edge of card; grip icon is a 3×2 dot grid SVG (⠿ equivalent), `text-gray-400` (D-09)
- Edit trigger: "Redigera" text button (Label scale, accent color) on card right edge — appears on hover/focus on desktop, always visible on mobile (D-07)

### WishItemCard (edit mode)
- Same card width; fields become `<input>` / `<textarea>` elements (D-07)
- Title input: required, full width, Heading scale text
- Price input: type="number", min=0, Label scale
- Product URL input: type="url", full width, Label scale
- Image URL input: type="url", full width, Label scale
- Note textarea: Body scale, 3 rows
- Save button: accent background, white text, "Spara" label (44px min height)
- Cancel button: ghost (transparent background, muted text), "Avbryt" label
- Delete button: destructive, text-only, "Ta bort önskemål" label — appears below Save/Cancel row (D-08)

### InlineDeleteConfirmation
- Replaces Delete button text with "Är du säker? [Ja, ta bort] [Avbryt]" inline (D-08)
- No modal — the confirmation appears in place of the Delete button
- "Ja, ta bort" uses destructive color (`text-red-600`)

### AddItemForm (inline expanded)
- Triggered by "+ Lägg till önskemål" button (D-03)
- Same field set as edit mode (D-04)
- Appears below the list when list has items, or replaces empty state CTA when list is empty
- Save button label: "Lägg till" (accent background)
- Cancel button: "Avbryt" (ghost)
- Validation: title required; show inline error "Titel krävs" in `text-red-600 text-sm` below the title input on submit attempt

### EmptyState
- Centered vertically in the page (D-14)
- Illustration: ⭐ emoji at 64px font size (no external image dependency)
- Heading (Display scale, 28px bold): "Din lista är tom"
- Body (Body scale, 16px regular): "Vad önskar du dig? Lägg till ditt första önskemål!"
- CTA button: accent background, "Lägg till önskemål", opens AddItemForm

### LoadingSkeleton
- Renders 3 placeholder cards while Firestore query resolves
- Each skeleton: same card dimensions as WishItemCard, `bg-[#E5D5CC]` animated with CSS `animate-pulse`
- No text content in skeleton — shape-only loading state

### DragAndDropList (`@dnd-kit/sortable`)
- `SortableContext` wraps the card list with `verticalListSortingStrategy`
- Each card wrapped in `useSortable` — drag via handle only (activator on grip element, not whole card)
- `DragOverlay` renders a ghost copy of the dragged card at 90% opacity during drag
- On `onDragEnd`: compute new fractional index via `fractional-indexing`, write only the moved item's `position` field (one Firestore write, D-10)
- Touch sensor enabled via `TouchSensor` with 8px activation distance, 250ms delay (mobile drag-and-drop, D-15)

## Responsive Breakpoints

| Breakpoint | Min width | Card layout | Page padding |
|------------|-----------|-------------|--------------|
| Mobile | 375px | Single column, full width minus 2×`md` (16px) padding | `px-4` (16px) |
| Tablet | 768px | Single column, max-width 640px centered | `px-6` (24px) |

**Note:** Both breakpoints use single-column card list per D-15. Cards may be slightly wider on tablet — achieved naturally by `max-w-2xl mx-auto` on the list container.

Drag handles must respond to touch events on mobile — `TouchSensor` is declared in DragAndDropList spec above.

**Source:** D-15 (375px mobile, 768px tablet). UI-02 requirement.

## Accessibility Contract

- Drag handle `<button>` has `aria-label="Dra för att ändra ordning"` and `role="button"`
- `SortableContext` list has `role="list"`; each card has `role="listitem"`
- Edit trigger button has `aria-label="Redigera [item title]"` (dynamic)
- Delete confirmation uses `role="alert"` to announce inline confirmation to screen readers
- Form fields use `<label htmlFor>` association (consistent with login/register pattern)
- Error messages rendered with `role="alert"` (consistent with login page pattern)
- Color contrast: text `#171717` on `#FFF0E8` background = 15.6:1 (WCAG AAA). Muted text `#6B7280` on `#FFF9F5` = 4.6:1 (WCAG AA).
