## Component Spec: Reserve Button

Three states derived from `isOthersPurchase` / `isOwnPurchase` logic in `ViewerWishItemCard`.

### State 1 — Not reserved by anyone (active, togglable)

```
Label:     "Jag tänker köpa detta"
Icon:      None (lighter visual weight than purchase button which uses a checkmark)
Classes:   mt-3 flex items-center gap-2 rounded-xl px-4 py-2 font-semibold text-sm
           min-h-[44px] transition-colors border
           border-dashed border-[#E5D5CC] bg-white hover:bg-[#FFF0E8] text-[#171717]
aria-label: "Reservera {item.title}"
disabled:  false
```

Rationale: Dashed border differentiates "intent" from the solid-border purchase button, conveying lighter commitment (from CONTEXT.md `<specifics>`). White background matches the un-purchased state of the purchase button.

### State 2 — Reserved by current user (active, togglable off)

```
Label:     "Du tänker köpa detta"
Icon:      Inline SVG calendar/bookmark (16×16) — OR omit icon, label is self-explanatory
Classes:   mt-3 flex items-center gap-2 rounded-xl px-4 py-2 font-semibold text-sm
           min-h-[44px] transition-colors border
           bg-[#F97316] hover:bg-[#EA6C0A] border-[#F97316] text-white
aria-label: "Avboka reservation för {item.title}"
disabled:  false
```

Clicking this state calls `onToggleReserved(itemId, itemTitle, false)` to un-reserve.

### State 3 — Reserved by someone else (disabled)

```
Label:     "Reserverad av {reserverName}"
Icon:      None
Classes:   mt-3 flex items-center gap-2 rounded-xl px-4 py-2 font-semibold text-sm
           min-h-[44px] transition-colors border
           opacity-50 cursor-not-allowed border-[#E5D5CC] bg-white text-[#6B7280]
aria-label: "Reserverad av {reserverName}"
disabled:  true
```

Mirrors `isOthersPurchase` exact pattern from `ViewerWishItemCard` purchase button.

### State 4 — Hidden (purchase is dominant)

When `isPurchased` is true (by anyone), the reserve button is NOT rendered at all.

Rationale: D-14 — purchase state is visually dominant. D-03 — when the reserver purchases, reservation is auto-cleared by the API. When someone else purchased, showing a reserve button would be confusing noise.

Condition for hiding: `if (isPurchased) return null` for the reserve button section.

## Interaction Contract

| Trigger | Behavior |
|---------|----------|
| Click reserve button (State 1) | Optimistic UI: immediately show State 2; call `onToggleReserved(itemId, itemTitle, true)`; on error revert + show error |
| Click reserve button (State 2) | Optimistic UI: immediately show State 1; call `onToggleReserved(itemId, itemTitle, false)`; on error revert + show error |
| Click reserve button (State 3) | No-op — button is `disabled` |
| Reserve button while `reserving === true` | Button disabled (`disabled={reserving}`) — same pattern as `toggling` on purchase button |
| Item becomes purchased (real-time) | Reserve button hidden — no transition needed |

### Optimistic update strategy

The component mirrors the existing `toggling` / `setToggling` pattern exactly:
- Local `reserving` boolean state
- `reserveError` string state
- `handleToggleReserve` async function wrapping `onToggleReserved` prop

No skeleton/spinner needed — the disabled state during the API call is sufficient feedback (existing pattern).

### Error copy

| Error scenario | Copy |
|----------------|------|
| Network / API error on reserve | "Något gick fel. Försök igen." |
| Network / API error on un-reserve | "Något gick fel. Försök igen." |
| Reservation already taken (409) | "Någon annan har redan reserverat detta." |

The 409 conflict case (race condition where two viewers click simultaneously) should surface as a distinct message since the outcome is meaningful — unlike a generic network error. The component should check `res.status === 409` and use the specific copy.
