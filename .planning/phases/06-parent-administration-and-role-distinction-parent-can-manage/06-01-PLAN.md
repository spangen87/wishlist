Create src/app/api/auth/set-parent-claim/route.ts. This route is called by /register after Firebase Auth sign-up — it sets `role: 'parent'` custom claim and writes the user profile.

Replicate the set-viewer-claim pattern exactly, changing only the role string:

```typescript
import 'server-only';
import { NextRequest, NextResponse } from 'next/server';
import { adminAuth, adminDb } from '@/lib/firebase/admin';
import { FieldValue } from 'firebase-admin/firestore';

export async function POST(request: NextRequest) {
  const body = await request.json().catch(() => ({}));
  const { idToken } = body as { idToken?: string };

  if (!idToken) {
    return NextResponse.json({ error: 'idToken required' }, { status: 400 });
  }

  let decodedToken;
  try {
    decodedToken = await adminAuth.verifyIdToken(idToken);
  } catch {
    return NextResponse.json({ error: 'Invalid token' }, { status: 401 });
  }

  const { uid, email } = decodedToken;
  await adminAuth.setCustomUserClaims(uid, { role: 'parent' });

  await adminDb.collection('users').doc(uid).set({
    uid,
    email: email ?? '',
    role: 'parent',
    createdAt: FieldValue.serverTimestamp(),
  });

  return NextResponse.json({ ok: true });
}
```

Note per D-03: the 'parent' claim is used ONLY for routing/dashboard layout. It does NOT grant extra Firestore access — parentUids[] on each wishlist document controls actual admin rights.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → set-parent-claim | Unauthenticated claim self-escalation |
| Firestore client SDK → wishlists | Client reads wishlists not in their parentUids/viewerUids |
| register-child caller → parentUids write | Caller forges viewerIdToken to inject arbitrary UID into parentUids |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01-01 | Elevation of Privilege | set-parent-claim route | mitigate | Route verifies idToken via adminAuth.verifyIdToken before setting claim — only the token owner's claim is elevated, never an arbitrary UID |
| T-06-01-02 | Spoofing | register-child viewerIdToken | mitigate | viewerIdToken decoded by adminAuth.verifyIdToken — invalid/forged tokens caught, UID never trusted from request body directly |
| T-06-01-03 | Elevation of Privilege | Firestore rules parentUids read | mitigate | Read rule uses `request.auth.uid in resource.data.parentUids` — requires auth and membership in the specific document's array, not a global claim |
| T-06-01-04 | Elevation of Privilege | items write rule | mitigate | isParent() reads get() on the parent wishlist doc — Firestore evaluates per-document membership, cannot be bypassed client-side |
| T-06-01-05 | Tampering | isParent helper — get() cross-doc read | accept | Firestore get() in rules is safe for per-document checks; composite query risk is mitigated by keeping direct field checks (`resource.data.parentUids`) for collection queries |
</threat_model>
